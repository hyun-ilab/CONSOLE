# Dieter Rams Philosophy for Interactive Web Design

> 핵심은 "브라운 레트로 UI"를 흉내 내는 것이 아니다.  
> **기능이 먼저 보이고, 장식은 사라지는 인터랙티브 시스템**으로 번역하는 것이다.

이 문서는 Dieter Rams의 디자인 철학을 웹 인터랙션, AI 보조 디자인, 접근성, 사용성 기준으로 옮기기 위한 실행 브리프다. 목표는 Rams의 제품 언어를 표면적으로 복제하는 것이 아니라, 그의 원칙을 사용자가 직접 조립하고, 비교하고, 줄이고, 평가하는 도구 경험으로 구현하는 것이다.

---

## 1. 핵심 방향

### 디자인 DNA

- **기능 우선**: 사용자가 무엇을 할 수 있는지가 첫 화면에서 보여야 한다.
- **절제된 시각 언어**: 흰색, 오프화이트, 검정, 회색, 제한된 포인트 컬러를 사용한다.
- **명확한 조작감**: 작은 원형 버튼, 얇은 라벨, 그리드, 다이얼, 스위치처럼 기능이 읽히는 형태를 쓴다.
- **설명 가능한 구조**: 점수, AI 제안, 오류, 상태 변화는 반드시 이유와 다음 행동을 함께 제공한다.
- **지속 가능한 인터페이스**: 유행하는 장식보다 오래 쓸 수 있는 레이아웃, 컴포넌트, 상태 설계를 우선한다.

### 참고할 오브젝트

- `SK4 radio-phonograph`
- `T3 pocket radio`
- `TP1`
- `606 Universal Shelving System`
- `ET66 calculator`

이 오브젝트들은 원본 이미지를 그대로 복제하기보다, **추상화한 SVG 또는 3D 오브젝트**로 재해석하는 것이 안전하다.

## 2. Rams 원칙의 웹 UI 번역

| Rams 원칙 | 웹 UI 번역 | 구현 기준 |
| --- | --- | --- |
| Innovative | 새로움은 장식이 아니라 문제 해결 방식에서 나온다. | WebGL, drag, AI는 기능이 명확할 때만 쓴다. "멋져 보여서" 넣는 효과는 금지한다. |
| Useful | 첫 화면부터 사용자가 조립, 비교, 평가를 할 수 있어야 한다. | 랜딩 카피보다 작업 캔버스를 먼저 보여준다. |
| Aesthetic | 미감은 비례, 정렬, 밀도, 여백에서 나온다. | 8px 계열 그리드, 제한 색, 얇은 라벨, 균일한 패널 폭을 유지한다. |
| Understandable | 조작 결과가 즉시 이해되어야 한다. | 스냅 위치, 단위, 상태, 오류, 점수 변화 이유를 바로 보여준다. |
| Unobtrusive | UI는 사용자의 구성물을 방해하지 않는다. | 패널은 필요할 때만 확장하고, hover 장식과 배지 과잉을 피한다. |
| Honest | 시스템 능력과 한계를 숨기지 않는다. | AI 제안에는 근거, 확신도, 수정 가능성을 붙인다. 가짜 정밀 점수는 금지한다. |
| Long-lasting | 유행 UI보다 오래 버틸 구조를 우선한다. | glassmorphism, 과한 3D, 일회성 hero 연출보다 안정적인 레이아웃을 쓴다. |
| Thorough | 사소한 상태까지 완결되어야 한다. | focus, disabled, loading, empty, error, mobile 상태를 모두 설계한다. |
| Environmentally-friendly | 웹에서도 가벼움과 지속 가능성을 본다. | 이미지/3D 자산 최소화, lazy load, reduced motion, 불필요한 autoplay 금지. |
| As little design as possible | 보이는 모든 것은 정보, 상태, 행동 중 하나를 맡아야 한다. | 장식 요소마다 "없애면 기능이 손상되는가?"를 통과해야 한다. |

## 3. 사용자 핵심 행위

1. **관찰한다**  
   사용자는 Rams 원칙, 오브젝트, 모듈 상태를 한 화면에서 읽는다. 첫 3초 안에 "내가 조립하고 평가하는 도구"임을 알아야 한다.

2. **조립한다**  
   사용자는 606식 레일과 bay에 모듈을 드래그하고 스냅한다. 핵심 행위는 꾸미기가 아니라 구조를 만드는 것이다.

3. **조정한다**  
   사용자는 다이얼, 스위치, 슬라이더로 크기, 밀도, 정렬, 대비, 라벨 노출 정도를 조절한다.

4. **드러낸다**  
   사용자는 SK4의 투명 뚜껑처럼 레이어를 열어 내부 구조, 원칙 근거, 점수 이유를 확인한다.

5. **줄인다**  
   사용자는 요소를 추가하는 만큼 제거도 해야 한다. "less but better"는 슬로건이 아니라 삭제와 정리 행위로 경험되어야 한다.

6. **비교한다**  
   사용자는 변경 전/후, 원칙 on/off, AI 제안 전/후를 비교하고 자기 선택을 되돌릴 수 있어야 한다.

7. **평가한다**  
   사용자는 결과물의 Rams score와 원칙별 근거를 보고 다음 수정 행동을 고른다.

## 4. 웹 인터랙션 설계

### 606 모듈러 빌더

- 선반, 캐비닛, 데스크 모듈을 레일 위에 배치한다.
- 모듈은 65cm/90cm bay 기준으로 스냅된다.
- 색상은 off-white, black, silver, beech 정도로 제한한다.
- 카드 나열보다 **조립 가능한 부품 시스템**으로 보여준다.

### 원칙별 제품 분해

- 10원칙을 누르면 같은 오브젝트가 다른 방식으로 강조된다.
- `Understandable` 선택 시 버튼, 다이얼, 라벨이 드러난다.
- `Honest` 선택 시 과장된 효과가 제거되고 근거가 보인다.
- 원칙 토글은 화면 색만 바꾸는 장치가 아니라, 정보 강조 방식을 바꾸는 장치여야 한다.

### Braun Control Panel

- T3/ET66에서 영감 받은 버튼 그리드, 슬라이더, 다이얼을 만든다.
- 조작은 빠르고 조용해야 한다.
- hover 장식은 최소화하고, 상태 변화는 명확하게 보여준다.

### Less But Better Meter

- 사용자가 요소를 추가할수록 다음 지표가 변한다.
  - `visual noise`
  - `usefulness`
  - `repairability`
- 장식을 더하면 점수가 오르는 구조가 아니라, 오히려 균형을 잃을 수 있는 구조로 설계한다.

### SK4 Transparent Lid

- 제품을 카드처럼 보여주는 대신, 뚜껑을 열어 내부 구조를 드러내는 인터랙션을 사용한다.
- "이해 가능성"은 설명 텍스트만이 아니라 구조를 드러내는 동작으로 구현한다.

## 5. 인터랙션별 존재 이유

| 인터랙션 | 존재 이유 | 사용 기준 |
| --- | --- | --- |
| Drag | 사용자가 구조를 몸으로 이해하게 한다. | 단순 장식 배치가 아니라 모듈과 레일의 관계를 배우는 행위여야 한다. |
| Snap | Rams식 정밀함을 경험시키는 핵심 피드백이다. | 자유 배치보다 제한된 좋은 위치를 제안한다. |
| Rotate | 물건을 보는 시점을 바꿔 이해 가능성을 높인다. | 3D 과시용 자동 회전은 피한다. |
| Switch | on/off의 즉각성을 보여준다. | 원칙 토글, 라벨 표시, 전원 상태처럼 이분 상태에만 쓴다. |
| Dial | Braun control language를 직접 조작으로 번역한다. | 연속값, 밀도, 밝기, 간격처럼 아날로그 감각이 있는 값에만 쓴다. |
| Reveal | SK4식 투명 뚜껑의 역할을 한다. | 내부 구조, 평가 근거, AI 판단 근거를 숨겼다 보여준다. |
| Principle Toggle | 같은 오브젝트를 다른 원칙으로 읽게 한다. | 전체 테마 변경이 아니라 관련 요소만 조용히 강조한다. |
| Less Meter | 많이 넣을수록 좋아지는 게임을 막는다. | 요소 수, 시각 소음, 유용성, 수리 가능성의 균형을 보여준다. |
| Compare | 사용자가 자기 판단을 갖게 한다. | AI 제안은 적용보다 비교 후 선택이 먼저다. |
| Undo/Reset | 실험을 안전하게 만든다. | 정직한 시스템은 되돌릴 권리를 항상 제공한다. |

## 6. 시각 디자인 기준

### 배경

- 거의 흰색 또는 아주 연한 회색을 사용한다.
- 어두운 유리 패널, 보라/파랑 그라디언트, 떠다니는 구체, 과한 bokeh, 의미 없는 particle은 사용하지 않는다.

### 타이포그래피

- Helvetica 또는 Inter 계열을 사용한다.
- 작은 사이즈, 넓은 여백, 명확한 계층을 유지한다.
- 과한 굵기와 과장된 display typography는 피한다.

### 컬러

- 회색 70%
- 흰색/검정 25%
- 포인트 컬러 5% 이하

포인트 컬러는 초록 on/off, 빨강 경고, 노랑 선택처럼 의미가 있을 때만 사용한다.

### 모션

- spring보다 linear 또는 ease-out을 우선한다.
- "기계가 정확히 움직인다"는 느낌을 준다.
- 필수 피드백 모션은 150-250ms 안에 끝낸다.
- reduced motion 설정에서는 위치 변화만 남기고 장식 모션은 끈다.

### 컴포넌트

- 카드 더미보다 패널, 레일, 슬롯, 라벨, 측정선, 모듈 조립감을 우선한다.
- 버튼, 다이얼, 스위치, 핸들은 기능이 눈에 보이는 형태여야 한다.

## 7. 접근성 및 사용성 기준

| 영역 | 기준 |
| --- | --- |
| 키보드 | 모든 drag/snap 행위에는 키보드 대체 조작이 있어야 한다. 방향키 이동, Enter 선택, Escape 취소, Ctrl+Z 되돌리기를 제공한다. |
| 포커스 | focus visible은 항상 보여야 한다. focus order는 화면의 읽기 순서와 같아야 한다. |
| 키보드 트랩 | 모달과 패널에서 keyboard trap을 만들지 않는다. |
| 대비 | 본문 텍스트는 최소 4.5:1, 큰 텍스트와 비텍스트 UI는 최소 3:1을 목표로 한다. |
| 타깃 크기 | 핵심 버튼과 핸들은 최소 24x24 CSS px, 터치 주 사용 영역은 44x44 px에 가깝게 잡는다. |
| 라벨 | 아이콘 단독 버튼에도 accessible name과 tooltip을 둔다. 숫자 슬라이더에는 단위와 현재 값을 붙인다. |
| 상태 | 저장, 스냅, 충돌, 비활성, 로딩, AI 분석 중, 점수 갱신 상태를 시각과 텍스트로 함께 알린다. |
| 오류 | 문제와 해결 행동을 같이 말한다. 빨간색만으로 오류를 전달하지 않는다. |
| 모바일 | 우측 패널은 하단 sheet로 바꾸고, 주요 조작은 엄지 영역 안에 들어오게 한다. 텍스트 겹침은 실패로 본다. |

사용성 평가는 NN/g 기준을 따라 다음 항목을 우선 검사한다.

- system status
- real-world mapping
- user control
- consistency
- error prevention
- recognition over recall

## 8. AI 디자인 원칙

AI는 메인 행위가 아니라 **조언 패널**이다. 사용자가 조립하고 판단하는 흐름을 밀어내서는 안 된다.

### 해야 할 것

- AI 제안에는 근거, 한계, preview, undo를 함께 제공한다.
- 제안은 한 번에 1-3개의 실행 가능한 수정으로 제한한다.
- 적용 전 비교할 수 있어야 한다.
- 불확실한 추천에는 "추정", "근거", "수정 가능"을 표시한다.
- 점수는 원칙별 근거와 다음 행동을 동반해야 한다.

### 하지 말아야 할 것

- "AI가 알아서 멋지게 만듭니다" 같은 마법형 문구를 쓰지 않는다.
- 채팅창을 메인 화면으로 두고 실제 조립 행위를 밀어내지 않는다.
- 불확실한 추천을 단정형으로 제시하지 않는다.
- 점수만 보여주고 이유를 숨기지 않는다.
- AI가 사용자의 배치를 자동으로 덮어쓰게 하지 않는다.
- 사람처럼 과도하게 말하는 AI 톤을 쓰지 않는다.

## 9. 카피 톤 규칙

### 기본 톤

- 조용하고 정확하게 말한다.
- "대단한 경험"보다 "정렬됨", "근거 있음", "줄일 수 있음"처럼 상태와 행동을 말한다.
- 한 문장에는 하나의 정보만 담는다.
- 가능하면 20자 안팎의 짧은 한국어를 쓴다.

### 버튼 문구

버튼은 명사보다 동사를 쓴다.

- `정렬`
- `줄이기`
- `비교`
- `근거 보기`
- `되돌리기`
- `스냅 해제`

### 점수 문구

나쁜 예:

> 82점

좋은 예:

> 82점. 라벨 대비와 모듈 간격은 좋지만, 장식 요소 2개가 목적을 설명하지 못합니다.

### 오류 문구

나쁜 예:

> 실패했습니다.

좋은 예:

> 이 폭에는 캐비닛이 들어가지 않습니다. 90cm bay로 옮기거나 폭을 줄이세요.

### AI 문구

나쁜 예:

> 제가 고쳤어요.

좋은 예:

> 제안 2개가 있습니다. 적용 전 비교할 수 있습니다.

### 금지어

- 혁신적
- 마법처럼
- 몰입감 넘치는
- 궁극의
- 아름다운 경험
- AI가 완성
- 알아서 최적화

### 허용어

- 유용함
- 근거
- 정렬
- 간격
- 대비
- 라벨
- 상태
- 조정
- 줄이기
- 비교
- 되돌리기

## 11. 게이트 체크

- CHECK: 한 화면 안에서 사용자의 다음 행동이 보인다.
- CHECK: 장식 요소를 제거했을 때 기능이 그대로라면 그 장식은 삭제했다.
- CHECK: drag로 가능한 일은 키보드로도 가능하다.
- CHECK: 점수와 AI 제안에는 반드시 이유와 되돌리기가 있다.
- CHECK: 모바일에서 텍스트, 버튼, 패널, 캔버스가 겹치지 않는다.
- CHECK: 원칙 토글은 화면 색만 바꾸지 않고 실제 정보 강조 방식을 바꾼다.
- CHECK: "less but better"가 슬로건이 아니라 실제 줄이기 행동으로 구현되어 있다.
- CHECK: AI는 사용자의 판단을 대체하지 않고 비교 가능한 제안을 제공한다.
- CHECK: 오류는 문제와 해결 행동을 함께 말한다.
- CHECK: reduced motion, focus visible, accessible name, tooltip이 빠지지 않았다.

## 12. 구현 MVP

1. **첫 화면**  
   606 스타일의 모듈러 그리드를 보여준다.

2. **우측 패널**  
   10 principles 토글을 둔다.

3. **중앙 캔버스**  
   Braun/Vitsœ에서 영감 받은 추상 제품 오브젝트를 배치한다.

4. **핵심 인터랙션**  
   drag, snap, rotate, reveal, switch, dial을 제공한다.

5. **결과 평가**  
   사용자가 만든 구성을 `Rams score`로 평가하고, 원칙별 근거와 다음 수정 행동을 보여준다.
